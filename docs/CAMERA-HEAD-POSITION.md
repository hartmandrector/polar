# Camera Head Position — Implementation Complete

## Overview

Second head orientation source for the GPS viewer: **Insta360 X5 camera quaternions** exported from Gyroflow. Complements the existing FlySight fused sensor (`SENSOR_fused_fusion.csv`) with an independent orientation measurement from the camera's IMU.

**Status: Phase 1–3 implemented.** Camera CSV loading, dual-source head rendering, mount offset controls, and sync-result.json persistence all working.

## Architecture

### Data Flow

1. **Manual load**: Click "Load Camera Head Position" → file picker → select gyroflow CSV → parsed with sync offset from `sync-result.json`
2. **Auto-load from URL**: When `?track=07-29-25/TRACK.CSV` loads, checks `public/07-29-25/sync-result.json` for `gyroflow` field → fetches CSV via Vite `/@fs/` from absolute path (e.g. OneDrive cloud storage)
3. **Save Config**: Writes `sync-result.json` to `public/<trackFolder>/` via Vite dev server POST `/api/sync-config`

### Data Priority
When both sources are available:
1. **Camera quaternion** → drives the head GLB model (priority)
2. **FlySight fused sensor** → fallback if no camera data
3. **Neither** → head model hidden

### Mount Offset
Camera orientation offset (misalignment) corrected via heading/pitch/roll sliders:
```
head_quat = camera_quat × inverse(mount_rotation)
```
Default: pitch = -20° (compensate forward camera tilt on selfie stick)

### Time Synchronization
- Camera CSV timestamps are relative (ms from video start, can be negative)
- `pipeline_time_ms = camera_timestamp_ms + syncOffsetMs`
- `syncOffsetMs` from `sync-result.json` (generated by `video-sync.ts`)
- Manual override via sync offset input in UI

### Interpolation
- Slerp between nearest camera quaternions at render time
- Works with 30fps (per-frame) or 1kHz (full export) data

## Open Questions

1. **Gyroflow quaternion convention**: What frame? Gravity-aligned + magnetic north? Need to test by loading data and comparing known orientations (standing upright, looking north, etc.)

2. **`org_quat` vs `stab_quat`**: The `org` (original) is the raw fused orientation. The `stab` (stabilized) is after Gyroflow applies stabilization. We want `org` — it represents actual camera orientation in space.

3. **Camera position relevance**: For the GPS viewer, we care about HEAD orientation (to position the head model). The camera's physical position on the helmet matters for accurate lever-arm correction but is secondary. The orientation offset (stick angle) is what matters most.

4. **Auto-detect sync**: Could we auto-detect the Gyroflow project file or sync-result.json in the same folder as the camera CSV? Would simplify the load flow.

5. **FlySight fused vs camera IMU quality**: Which IMU is better? FlySight has a dedicated 9DOF IMU. Insta360 X5 IMU is optimized for video stabilization (high gyro rate, may have different accel noise characteristics). Visual comparison will reveal this.
